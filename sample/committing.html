GitHub, Atom, and Markdown for Creative Writers

Making the First Commit

Now that the repository is open in Atom, the next step is to create the first commit. Much like the glossy diction of good science-fiction, some jargon must be defined. The above picture shows the Git tab in Atom. Select Git in the lower right corner of the screen to bring it up. At the bottom of this tab, it reads "Not on branch." A branch represents, in the software development cycle, one collaborator's extant version which may eventually be merged, or incorporated, into the main branch by a pull request, also mentioned on that tab. For the purposes of creative writing, this may seem irrelevant, but the next few steps are necessary in order to initiate the main branch, beginning with creating a file in Atom.

On the far left pane in Atom, the Project tab contains the local clone of the repository. It contains nothing aside from the .git folder, which should be ignored for these purposes. Right-click (on Windows) or control-click (on macOS) on the name of the repository. Select "New Folder."

A bar will appear. Name this folder and it will appear in the repository. Right- or control-click the folder and select "New file."

The name bar will fill in the folder name where it's created. Enter a name and add .md as a file extension. Familiar file extensions include .docx or .pdf for Microsoft Word and Adobe Documents, respectively, while .md stands for Markdown. Markdown is a formatting language which will be introduced in full later in this guide. This file will open in one of the center panes in Atom.

Enter some text in the file. Notice the dot that appears to the right of the file name. This indicates that the file needs to be saved using control-s (on Windows) or command-s (on macOS). This only saves to the repository on the local computer. Committing presents batches of file changes to the repository on GitHub.

After saving the file, return to the Git tab on the right. The new file will appear under Unstaged Changes. These are the locally modified files which must first be staged by selecting "Stage All." This prepares Git for the changes to the repository without making them part of the record. If changes are staged, they must either be unstaged using "Unstage All" or committed.

Once changes have been staged, write a comment for the commit. Whether coding or writing, sometimes older commits need to be revisited. Making a habit of writing descriptive comments will aid the ongoing process of revising and editing.

Right now, the commit button reads "Create detached commit." After this first commit is published, the main branch will exist for all future commits. Click the button to create the commit.

The next step is usually to push the commit to the branch. However, because the main brain doesn't yet exist outside of our local copy, there are a couple more steps. In the bottom right corner of Atom, select "Fetch." This will open the web browser and ask for the GitHub login and an authorization as it did when installing GitHub Desktop.

For the final step, open GitHub Desktop. The first commit will appear in history. To publish the first commit, select "Publish" to the right of "Current Branch." Check the repository on GitHub.com. Instead of the commands that were there when the repository was created, the first commit should appear along with the folder and file.

The process can be completd entirely within Atom from this point forward. The procedure goes:

  1. On opening Atom, "Fetch" or "Pull" at the bottom right to ensure your cloned repository has the latest version. This becomes more relevant when using multiple machines or with collaborators.
  2. Work on files. Do another test edit on the file to practice committing.
  3. Stage the changes.
  4. Write a comment and commit.
  5. Following that, select "Push," which will be in the place of "Fetch."

These few steps will quickly become a reliable habit. From here on, the focus in Atom will be on writing, beginning with customizing it for this purpose.
